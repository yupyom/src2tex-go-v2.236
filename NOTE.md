# 雑多なノート — src2tex-go の設計について

本ドキュメントでは、src2tex-go の設計の背景を記録します。

## 作成のきっかけ

オリジナル版の src2tex は、30年前の天野先生との出会いまで遡ります。
そもそもは講義ノートをどのように見れば良いかの質問に行ったのがきっかけです。私は当時からMacintoshユーザーでしたので、TeXには馴染みがなく、新鮮でした。

当時もMacで使えるように移植したいと思い、先生からソースを分けてもらいましたが、とうとう実現はできず。それから月日は流れ、紆余曲折あって、ちょっとしたプログラムを作っているときに、ドキュメンテーションをどうしようと思ったときに思い出しました。

もう先生も退官なさって久しく、ご連絡先もわからなくなっていました。当時の研究室のホームページも閉鎖されており、古いLinuxのアーカイブなどから探してきて、移植しはじめたのがきっかけとなります。

## 問題意識

オリジナル版が作られた時期が古く、Kernighan & Ritchie 時代の古い C スタイルで記述されていたことと、当時の TeX スタイルのプリアンブルや、EUC-JPの利用が前提となっていたので、現代的な環境で使うには少し難がありました。

私の記憶では、当時、先生は Solaris や Linux を使っていましたが、C言語は環境によってはコンパイル環境を揃えるのがやや大変（当時に比べたらだいぶ良くなりましたが）という事情もありました。

そこでポータビリティを持たせるために、Go言語ベースにポーティングすることと、LaTeX用プリアンブルを分離する設定ファイル方式をとることとしました。

## なぜ goyacc などのパーサジェネレータを使わなかったのか

Go には goyacc をはじめ、ANTLR や tree-sitter バインディングなどのパーサジェネレータ／レキシカルアナライザが存在します。しかし src2tex-go では独自のスキャナとエミッタで実装しました。いくつかの理由がありますが、最も本質的なのは、 **このタスクの特性がパーサジェネレータの得意分野と合致しない** という点です。

### 「解析」ではなく「分類」が目的

goyacc/ANTLR/tree-sitter などのパーサジェネレータは、**構文木（AST）を構築する**ことを主目的としています。つまり `if (x > 0) { return x; }` を構文的に理解し、「条件式」「ブロック」「return文」といった **構造** を認識します。

しかし src2tex に必要なのは構文木ではなく、入力テキストを以下の **5種類に分類** するだけです：

| TokenKind | 目的 |
|-----------|------|
| `TokenCode` | そのまま等幅フォントでエスケープ出力 |
| `TokenKeyword` | ボールドで出力 |
| `TokenLineComment` | セリフフォント＋TeXパススルー |
| `TokenBlockComment` | 同上 |
| `TokenTeX` | 完全にverbatim出力 |

文の構造、式の優先順位、スコープ — いずれも不要です。これは
**字句解析（lexical analysis）のサブセット** であり、構文解析（parsing）ですらありません。

### 25言語を1つの文法で扱うコスト

goyacc を使うなら言語ごとに `.y` ファイル（文法定義）が必要です。25言語分の文法を書くのは **膨大なコスト** です。しかも src2tex が必要とする情報は「コメントの開始/終了トークン」「キーワード一覧」「文字列リテラルの区切り文字」だけです。

今回のテーブル駆動設計（`LangDef` 構造体）は、1つのスキャナで **データだけを差し替えて** 全言語に対応できます。これは goyacc ではかえってコストがかかると判断して、独自に解析する形をとりました。

### ステートマシンの単純性

スキャナの本質は `stateCode → stateLineComment → stateCode` のような **有限オートマトン** です。約500行のコードで文字列リテラルのスキップ、複合言語のコンテキストスタック（HTML → CSS/JS/PHP 切り替え）、ネスト可能なブロックコメント（Pascal の `{ }`）まで処理しています。

goyacc を導入しても、この部分の複雑さはほとんど変わらず、むしろ **抽象化のオーバーヘッド** が増えるだけです。

### 原典（オリジナル src2tex）との類似性

1990年代のオリジナル src2tex もフラグベースの手書きステートマシンでした。
Go移植にあたって「フラグ管理 → 明示的ステートマシン」という近代化を行いましたが、 **処理モデルそのものは忠実に保つ** 判断をしています。パーサジェネレータを導入すると、出力の微妙な差異のデバッグが難しくなると思います。

## 変換プログラムとしてのポイント

src2tex は本質的に「ソースコード → LaTeX」の **変換パイプライン** です。HTML のレンダリングエンジンや Markdown → HTML 変換ツールと同じカテゴリに属します。

この観点から、設計上の主要なポイントを整理します。

### 二段パイプライン分離（Scanner → Emitter）

```
src → [Scanner] → []Token → [Emitter] → LaTeX (.tex) 
```

これは markdown-it（Markdown → HTML）の `parse() → render()` と同じ設計です。
重要なのは **トークン列がインターフェースとなっている** こと。
スキャナは「何がコメントか」を判定し、エミッタは「コメントをどう LaTeX に変換するか」
だけに集中します。

この分離により、たとえば将来 HTML エミッタを追加する場合、
スキャナは一切変更不要です。

### TeX の特殊文字エスケープ対応

HTML レンダリングで `<` → `&lt;` のエスケープが最重要なように、src2tex では **TeX の特殊文字エスケープ** が変換の核心です。

```
// コードコンテキスト（等幅フォント）
' ' → {\tt\mc \ }
'#' → {\tt\#}
'{' → {\tt\char'173}

// コメントコンテキスト（セリフフォント）
'_' → \_
'$' → \$
```

**同じ文字でもコンテキスト（コード vs コメント）でエスケープが異なる** ことが、トークン分類が必要な理由であり、単純な正規表現置換などでは実現が困難な部分です。

### 行指向レンダリング

TeX は **行の概念が弱い** 文法です。src2tex はソースコードの行構造を TeX に強制するために独特の出力パターンを採用しています：

```tex
\noindent
\mbox{}{\tt\mc \ }{\tt\mc \ }{\tt f}{\tt o}{\tt r}  % インデント付きコード行
```

`\noindent` + `\mbox{}` の組み合わせで各行を独立段落として扱い、`\hfill` で空行を表現する — この「TeX の段落モデルとソースコードの行モデルの、いわばインピーダンスミスマッチをどう解消するか」が、最大の設計課題でした。

### フォントモード切り替え

src2tex の出力が美しい理由は、**コードは等幅（`\tt`）、コメントはセリフ（`\rm`）** という書体の使い分けです。コメントマーカー部分（`//` や `#`）だけは `{\tt /}` で等幅に戻すという細やかなフォント制御が、最終出力となるPDFの可読性を大きく向上させています。

### TeX パススルー

```c
/* {\centerline{\epsfbox{figure.eps}}} */
```

コメント内の `{\ ... }` をそのまま TeX として出力する仕組みは、単なる文法変換ツールを超えて、**ソースコード文書に数式・図版・任意の TeX 命令を埋め込める** という表現力を生んでいます。これは原典 src2tex の最も独創的な設計判断であり、Go 移植でも忠実に再現しました。

なお、オリジナル版では `{\ }` マーカーだけでなく、コメント内の `$...$` も暗黙的に TeX パススルーとして扱っていました。これはコメント開始時にバッファ全体を先読みスキャンし、`$` が含まれていればコメント全体を TeX モード（`RMFlag=1`）に切り替えるという粗い方式でした。

v2.236 では、この先読みスキャンのアプローチを踏襲せず、`$...$` の範囲だけを精密にパススルーする方式に再設計しました。これにより、コメント内の `$` がない部分は従来通り `commentTextEscape` でエスケープされ、`$...$` 内だけがそのまま TeX 数式として出力されます。トークンベースの設計と整合する、より予測可能な動作になったと考えています。

また、`{\ ... }` パススルー内容が複数行にまたがる場合の改行処理も、v2.236 で初期実装では欠落していました。旧バージョンでは1文字ずつ処理する中で自然に `\noindent\mbox{}` が挿入されていましたが、v2.236 のトークンベース設計ではパススルー内容を一括で扱うため、明示的な改行処理メソッド `writePassthroughWithLineBreaks` を導入して解決しました。

## まとめ

こうして書くと、結局のところ、ほとんどがオリジナル版の設計思想を色濃く残した形になっています。

| 観点 | 設計判断 |
|------|----------|
| パーサ選択 | 字句分類だけで十分 → 手書きステートマシン + テーブル駆動 |
| 多言語対応 | 文法記述ではなくデータ（LangDef）で25言語を統一処理 |
| 変換の核心 | コンテキスト依存の TeX エスケープ |
| 最大の課題 | TeX の段落モデルとソースコードの行モデルの橋渡し |
| 独自の強み | コメント内 TeX パススルーによる表現力 |

要するに、 **「構文を理解する」のではなく「表面的な字句パターンを分類して、文脈に応じた TeX 出力に変換する」** というタスクの本質を見極めた結果、パーサジェネレータではなく独自のスキャナ＋エミッタという軽量なアーキテクチャが最適解になったということでした。

何かの参考になりましたら幸いです。
